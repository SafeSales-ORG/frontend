# NIP.md — SafeSale Nostr Event Specification

> This document defines every Nostr event kind, tag, and content schema used by SafeSale. It is the contract between the frontend, the backend, and any third-party Nostr client that wants to interoperate with SafeSale data.
>
> When adding a new kind or changing an existing schema, update this file in the same PR.

---

## Relays

SafeSale publishes to and reads from a curated relay set defined in `src/lib/appRelays.ts`. The hackathon set includes at minimum:

- `wss://relay.damus.io`
- `wss://nos.lol`
- `wss://relay.nostr.band`
- `wss://relay.primal.net`

A SafeSale-operated relay (`wss://relay.safesale.app`) is planned post-MVP for archival of escrow and dispute events.

---

## Identities

| Role | Identity | How it's obtained |
|---|---|---|
| Seller | Long-lived Nostr keypair | Auto-generated in browser on first visit, stored in `localStorage`. May be replaced by an existing key via extension / nsec / NIP-46 remote signer. |
| Buyer | One-time Nostr keypair per order | Auto-generated by the SafeSale backend at checkout. Returned to the browser, also stored in the order record so the mint can verify P2PK signatures. The buyer may later replace it with their own key if they choose. |
| Mediator (SafeSale admin) | Single well-known Nostr pubkey | Hard-coded as `MEDIATOR_PUBKEY` in `src/lib/safesale.ts`. Used to sign dispute resolutions. Compromise of this key = full mediation compromise; rotate via a published replacement event (kind `30078`, `d=mediator-key-rotation`). |

## Event kinds

| Kind | Source | Purpose | Standard |
|---|---|---|---|
| `0` | Seller, buyer | Profile metadata (name, picture, about) | NIP-01 |
| `1059` | Seller, buyer | Gift-wrapped DM (encrypted buyer↔seller chat per order) | NIP-17 / NIP-59 |
| `30018` | Seller | Marketplace product listing | NIP-15 |
| `30402` | Seller (alternative to 30018) | Classified listing (used if 30018 is too rigid) | NIP-99 |
| `33888` | Backend (escrow service) | **SafeSale order event** — custom replaceable | Custom (see below) |
| `33889` | Mediator | **SafeSale dispute resolution event** — custom replaceable | Custom (see below) |
| `1985` | Buyer | Verified review / label attached to a completed order | NIP-32 (labels) |
| `30078` | Mediator | App-specific data (mediator key rotation, app config) | NIP-78 |

**Why custom kinds 33888 and 33889:** existing NIPs do not cover *escrow orders* or *signed dispute resolutions tied to a specific Cashu token*. We register these in the `30000–39999` replaceable range so they can be updated by the same author (the backend's escrow service key for orders; the mediator key for resolutions).

## Kind 30018 — Product Listing (NIP-15)

Published by the seller. Replaceable; the `d` tag is the listing's UUID.

### Tags

| Tag | Required | Example | Notes |
|---|---|---|---|
| `d` | yes | `7f3a9c` | Listing UUID. The `safesale.app/buy/<d>` URL maps directly. |
| `title` | yes | `Vintage denim jacket` | |
| `summary` | recommended | `Used twice, size M, no tears` | Short description |
| `image` | repeatable | `https://nostr.build/i/abc.jpg` | At least 1, max 4 |
| `price` | yes | `["price", "8500", "NGN"]` | Three-element form: amount, currency. NGN is primary. |
| `price_sats` | optional | `["price_sats", "12500"]` | Cached sats equivalent at publish time; UI recomputes live. |
| `stock` | optional | `["stock", "1"]` or `["stock", "out"]` | If absent, treated as `1`. |
| `category` | optional | `["category", "fashion"]` | One of: `fashion`, `electronics`, `beauty`, `food`, `art`, `digital`, `other` |
| `t` | repeatable | `["t", "denim"]` | Free-form tags for search (NIP-12 single-letter standardised filter) |
| `published_at` | optional | unix seconds string | |
| `delivery` | optional | `["delivery", "pickup", "lagos"]` or `["delivery", "shipping"]` | Hints to the buyer |
| `r` | optional | `["r", "https://safesale.app/buy/7f3a9c"]` | Canonical SafeSale URL |

### Content

Long-form description in Markdown. May reference images by URL (already in tags) for inline rendering.

### Example

```json
{
  "kind": 30018,
  "pubkey": "<seller-npub-hex>",
  "created_at": 1716240000,
  "tags": [
    ["d", "7f3a9c"],
    ["title", "Vintage denim jacket"],
    ["summary", "Used twice, size M, no tears"],
    ["image", "https://nostr.build/i/abc.jpg"],
    ["image", "https://nostr.build/i/def.jpg"],
    ["price", "8500", "NGN"],
    ["price_sats", "12500"],
    ["stock", "1"],
    ["category", "fashion"],
    ["t", "denim"],
    ["t", "vintage"],
    ["delivery", "shipping"]
  ],
  "content": "Worn twice for photoshoots. Selvedge denim, original buttons. Smoke-free home.",
  "sig": "..."
}
```

## Kind 33888 — SafeSale Order (custom)

Published by the **SafeSale escrow-service key** (held by the backend). Replaceable per order — the `d` tag is the order token. The event reflects the *current* state of the order; UIs should not maintain their own state machine.

### Tags

| Tag | Required | Example | Notes |
|---|---|---|---|
| `d` | yes | `ord_xk29mp` | Order token. Also the URL slug at `safesale.app/order/<d>`. |
| `status` | yes | `["status", "payment_locked"]` | One of: `pending_payment`, `payment_locked`, `shipped`, `delivered`, `released`, `disputed`, `resolved`, `refunded`, `expired` |
| `a` | yes | `["a", "30018:<seller-pubkey>:7f3a9c"]` | NIP-23 coordinate of the listing being purchased |
| `p` | yes, twice | `["p", "<seller-pubkey>", "", "seller"]`, `["p", "<buyer-pubkey>", "", "buyer"]` | Both parties |
| `amount` | yes | `["amount", "8500", "NGN"]` | NGN paid by buyer |
| `amount_sats` | yes | `["amount_sats", "12500"]` | Sats minted into the Cashu token |
| `mint` | yes | `["mint", "https://mint.nutshell.cash"]` | Cashu mint URL |
| `cashu_token_hash` | yes | `["cashu_token_hash", "<sha256>"]` | SHA-256 of the locked Cashu token. **Never publish the token itself** — it's bearer money. |
| `p2pk_pubkey` | yes | `["p2pk_pubkey", "<buyer-one-time-pubkey>"]` | The Cashu NUT-11 unlock pubkey |
| `tracking` | optional | `["tracking", "DHL", "1234567890"]` | Carrier + tracking number |
| `shipped_at` | optional | `["shipped_at", "1716340000"]` | Unix seconds |
| `delivered_at` | optional | `["delivered_at", "1716440000"]` | Unix seconds |
| `released_at` | optional | `["released_at", "1716540000"]` | Unix seconds |
| `auto_release_at` | yes if `status=shipped` | `["auto_release_at", "1716944800"]` | Unix seconds — 7 days after `shipped_at` |
| `dispute_id` | optional | `["dispute_id", "dsp_xyz"]` | If a dispute has been opened, links to the 33889 event |
| `bitnob_account` | optional | `["bitnob_account", "0124558947", "Wema Bank"]` | The virtual account the buyer paid into. Helpful for receipts; may be omitted for privacy. |

### Content

JSON-encoded structured data not suitable as tags:

```json
{
  "delivery_address": "[encrypted with seller pubkey via NIP-44]",
  "buyer_contact": "[encrypted with seller pubkey via NIP-44]",
  "buyer_email_or_sms_hint": "j***@gmail.com",
  "version": 1
}
```

The delivery address and contact are **NIP-44 encrypted to the seller's pubkey** so only the seller (and the mediator, who is also `p`-tagged) can decrypt. The escrow service publishes the event but does not include plaintext PII.

## Kind 33889 — Dispute Resolution (custom, signed by mediator)

Published by the **mediator key**. Replaceable per dispute — `d` tag is the dispute ID. The latest version reflects the current resolution state.

### Tags

| Tag | Required | Example | Notes |
|---|---|---|---|
| `d` | yes | `dsp_xyz` | Dispute ID |
| `e` | yes | `["e", "<order-event-id>", "", "order"]` | Pointer to the 33888 order event |
| `a` | yes | `["a", "33888:<escrow-service-pubkey>:ord_xk29mp"]` | Replaceable coordinate of the order |
| `p` | yes, three | `["p", "<seller>"], ["p", "<buyer>"], ["p", "<mediator>"]` | All parties involved |
| `status` | yes | `["status", "resolved"]` | One of: `open`, `direct_resolution`, `escalated`, `evidence_requested`, `resolved` |
| `reason` | yes | `["reason", "wrong_item"]` | Buyer's initial reason: `wrong_item`, `damaged`, `not_as_described`, `not_received`, `return_request` |
| `evidence` | repeatable | `["evidence", "<blossom-url>", "buyer"]` or `["evidence", "<url>", "seller"]` | Each piece tagged with the party that uploaded it |
| `outcome` | yes if `status=resolved` | `["outcome", "split", "60"]` | One of: `release_seller`, `refund_buyer`, `split <buyer_pct>`, `extended_review` |
| `released_sats` | yes if outcome `release_seller` or `split` | `["released_sats", "7500"]` | Sats released to the seller |
| `refunded_sats` | yes if outcome `refund_buyer` or `split` | `["refunded_sats", "5000"]` | Sats refunded to the buyer |
| `mediator_sig` | yes if `status=resolved` | `["mediator_sig", "<schnorr-sig>"]` | Mediator's signature over the canonical resolution payload (defensive — the event itself is already signed, but downstream consumers can verify without holding the full event) |

### Content

Free-form Markdown explaining the mediator's reasoning. Publicly readable, permanently archived. Example:

```
After reviewing the original listing, the buyer's photos of the received item, and the
seller's response, the item shipped does not match the description in two ways:
the size is M (listed as L) and there is a 4cm tear on the left sleeve not shown in
the listing photos.

The buyer is awarded a 60% refund (₦5,100). The seller retains 40% (₦3,400) to cover
the cost of shipping and a partial value for the item which the buyer keeps.

This decision is final and signed by mediator npub1...xyz.
```

## Kind 1985 — Verified Review (NIP-32 label)

Published by the **buyer** after a successful completion (or after a resolved dispute, if the resolution permits). Used to build the seller's reputation.

### Tags

| Tag | Required | Example | Notes |
|---|---|---|---|
| `L` | yes | `["L", "safesale.review"]` | Label namespace |
| `l` | yes | `["l", "5", "safesale.review"]` | Star rating 1–5 |
| `p` | yes | `["p", "<seller-pubkey>"]` | The seller being reviewed |
| `e` | yes | `["e", "<order-event-id>", "", "order"]` | The 33888 order this review attests to |
| `a` | yes | `["a", "33888:<escrow-service-pubkey>:ord_xk29mp"]` | Replaceable coordinate of the order |
| `verified_by` | yes | `["verified_by", "<escrow-service-pubkey>"]` | Buyers may sign reviews; the escrow service co-signs by referencing it in the order event |

### Content

The review text. Should be short (under 500 chars). No HTML, plain text + emoji.

### Validation rule

A SafeSale UI MUST verify that:

1. The review's `e` tag points to an order event where `status=released` or `status=resolved`
2. The reviewing pubkey matches the `p[role=buyer]` tag on that order event
3. Only one review per order per direction (one buyer→seller, optionally one seller→buyer)

Reviews that fail this check are silently dropped — they are not "verified."

## Kind 1059 — Encrypted DMs (NIP-17 / NIP-59 gift-wrap)

Used for **all buyer ↔ seller communication** within an order. Also used when the mediator requests evidence.

- Inner kind: `14` (chat message)
- Outer kind: `1059` (gift wrap)
- Encrypted with NIP-44
- Each message MUST include an `a` tag pointing to the order coordinate so clients can filter messages by order

### Inner kind 14 tags (within the gift wrap)

```json
{
  "kind": 14,
  "content": "Hi, I've shipped your order today via DHL — tracking 1234567890",
  "tags": [
    ["p", "<recipient-pubkey>"],
    ["a", "33888:<escrow-service-pubkey>:ord_xk29mp", "", "order"]
  ]
}
```

## Filter recipes

Queries used by the frontend, included here so the backend (when it indexes) knows what to optimize.

### A seller's active listings

```json
{ "kinds": [30018], "authors": ["<seller-pubkey>"], "limit": 50 }
```

### A seller's recent orders

```json
{ "kinds": [33888], "authors": ["<escrow-service-pubkey>"], "#p": ["<seller-pubkey>"], "limit": 50 }
```

### All resolved disputes (admin audit)

```json
{ "kinds": [33889], "authors": ["<mediator-pubkey>"], "#status": ["resolved"] }
```

### Reviews of a seller

```json
{ "kinds": [1985], "#p": ["<seller-pubkey>"], "#L": ["safesale.review"] }
```

### Buyer's order history (when the buyer has a persistent key)

```json
{ "kinds": [33888], "#p": ["<buyer-pubkey>"] }
```

## Security & validation rules (MUST)

1. **Author filtering on trust-sensitive events.** Always filter resolution queries by `authors: [MEDIATOR_PUBKEY]`. Never trust a kind `33889` event from any other author. Same for kind `33888` — only trust events from the official escrow service key.
2. **Verify P2PK conditions before releasing.** Cashu mints enforce P2PK at the protocol level, but the UI MUST also display the unlock pubkey to the buyer before they sign a release.
3. **Reject non-replaceable updates to `d`-tagged events.** If a kind `33888` event with a given `d` value already exists with a later `created_at`, ignore older ones.
4. **Sanitize listing fields before rendering.** `image`, `r`, `delivery` tags carry user-controlled strings. Always run them through `sanitizeUrl()` / `sanitizeCssString()` before interpolating.
5. **Never publish raw Cashu tokens to Nostr.** They are bearer money; anyone who sees the event has the money. Only publish the SHA-256 hash.
6. **NIP-44 encrypt all PII.** Delivery addresses and buyer contact info inside kind `33888` content MUST be NIP-44 encrypted to the seller (with the mediator as a secondary recipient).

## Versioning

This document is at **v1.0** as of the initial scaffold. Breaking changes bump the major version. Additions of optional tags are minor versions. The current version is also encoded in the `content.version` field of kind `33888` events.
